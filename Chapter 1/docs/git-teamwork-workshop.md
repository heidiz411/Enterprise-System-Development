# Workshop: การทำงานร่วมกับทีมด้วย Git CLI

> **ระดับ:** ผู้เริ่มต้น
> **รูปแบบ:** จับคู่ทำงาน 2 คนต่อ 1 กลุ่ม
> **เวลาที่ใช้:** ประมาณ 90–120 นาที
> **โจทย์:** สร้างและแก้ไขไฟล์ `README.md` ที่มีข้อมูลส่วนตัวของสมาชิกแต่ละคน โดยใช้ branch แยกกัน แล้วนำมารวม (merge) เข้าด้วยกัน

---

## วัตถุประสงค์การเรียนรู้

เมื่อทำ workshop นี้จบ นักศึกษาจะสามารถ:

1. ตั้งค่า Git บนเครื่องตัวเองได้ถูกต้อง
2. สร้าง repository บน GitHub และเชิญเพื่อนเข้าร่วมเป็นทีม
3. สร้าง **branch** ของตัวเองเพื่อทำงานโดยไม่กระทบงานของเพื่อน
4. ใช้คำสั่ง `add` → `commit` → `push` ในการบันทึกและส่งงานขึ้น GitHub
5. รวมงานของสองคน (merge) และ **แก้ไข merge conflict** เมื่อทั้งคู่แก้ไฟล์เดียวกัน

---

## สิ่งที่ต้องเตรียมก่อนเริ่ม

- [ ] ติดตั้งโปรแกรม **Git** บนเครื่อง (ตรวจสอบด้วยคำสั่ง `git --version`)
- [ ] มีบัญชี **GitHub** ทั้งสองคน (สมัครฟรีที่ https://github.com)
- [ ] โปรแกรมแก้ไขข้อความ เช่น VS Code, Notepad++ หรือ nano
- [ ] ในกลุ่มแบ่งหน้าที่ชัดเจน: ใครเป็น **คนที่ 1 (Owner)** และใครเป็น **คนที่ 2 (Member)**

> 💡 **เคล็ดลับ:** ในแต่ละขั้นตอนจะมีป้ายกำกับว่า **[คนที่ 1]**, **[คนที่ 2]** หรือ **[ทั้งคู่]** ให้ทำตามหน้าที่ของตัวเอง

## ขั้นตอนที่ 0 — ตั้งค่า Git ครั้งแรก **[ทั้งคู่]**

ทำครั้งเดียวต่อเครื่อง เพื่อให้ Git รู้ว่าใครเป็นคนแก้ไขงาน (จะไปแสดงในประวัติ commit)

```bash
git config --global user.name "ชื่อจริง ของคุณ"
git config --global user.email "อีเมลที่ใช้สมัคร GitHub"
```

ตรวจสอบว่าตั้งค่าถูกต้อง:

```bash
git config --global --list
```

> ⚠️ อีเมลควรเป็นอีเมลเดียวกับที่สมัคร GitHub เพื่อให้ระบบนับ commit ให้ถูกคน

---

## ขั้นตอนที่ 1 — สร้าง Repository ร่วมกัน **[คนที่ 1]**

คนที่ 1 ทำหน้าที่สร้างบ้านส่วนกลางให้ทีม

1. เข้า https://github.com แล้วกดปุ่ม **New** (สร้าง repository ใหม่)
2. ตั้งชื่อ เช่น `team-profile-workshop`
3. เลือก **Public**
4. ติ๊กถูกที่ **Add a README file** (เพื่อให้มี branch `main` พร้อมใช้งาน)
5. กด **Create repository**

---

## ขั้นตอนที่ 2 — เชิญเพื่อนเข้าทีม **[คนที่ 1]**

1. เข้าไปที่ repo ที่เพิ่งสร้าง → แท็บ **Settings**
2. เมนูซ้าย เลือก **Collaborators**
3. กด **Add people** แล้วใส่ **username GitHub ของคนที่ 2**
4. คนที่ 2 จะได้รับอีเมลเชิญ → กดยอมรับ (Accept invitation)

> ถ้าไม่เชิญขั้นตอนนี้ คนที่ 2 จะ `push` ขึ้น repo ไม่ได้

---

## ขั้นตอนที่ 3 — Clone repository ลงเครื่อง **[ทั้งคู่]**

ทั้งสองคนดึงสำเนา repo ลงมาที่เครื่องตัวเอง

1. ที่หน้า repo กดปุ่มสีเขียว **Code** → คัดลอก URL (เลือกแบบ HTTPS)
2. เปิด Terminal แล้วพิมพ์ (เปลี่ยน URL เป็นของกลุ่มตัวเอง):

```bash
git clone https://github.com/USERNAME/team-profile-workshop.git
cd team-profile-workshop
```

ตรวจสอบว่าอยู่ที่ branch `main`:

```bash
git status
```

---

## ขั้นตอนที่ 4 — สร้าง branch ของตัวเอง **[ทั้งคู่]**

แต่ละคนสร้าง branch ส่วนตัว **อย่าทำงานบน `main` โดยตรง**

**คนที่ 1** พิมพ์:

```bash
git checkout -b member-a
```

**คนที่ 2** พิมพ์:

```bash
git checkout -b member-b
```

> คำสั่ง `git checkout -b ชื่อbranch` = สร้าง branch ใหม่ และย้ายตัวเองเข้าไปทำงานใน branch นั้นทันที
> ตรวจสอบว่าอยู่ branch ถูกต้องด้วย `git branch` (branch ปัจจุบันจะมีเครื่องหมาย `*` นำหน้า)

---

## ขั้นตอนที่ 5 — แก้ไขไฟล์ README.md **[ทั้งคู่]**

เปิดไฟล์ `README.md` ด้วย editor ที่ถนัด แล้ว **แต่ละคนเพิ่มเฉพาะส่วนของตัวเอง** ต่อท้ายไฟล์

**คนที่ 1** เพิ่มข้อความนี้ (แก้ข้อมูลเป็นของตัวเอง):

```markdown
## สมาชิกคนที่ 1: [ชื่อ-นามสกุล]

- 🎓 รหัสนักศึกษา: 6xxxxxxxx
- 💻 ภาษาโปรแกรมที่ชอบ: Python
- 🎯 งานอดิเรก: เล่นเกม, อ่านหนังสือ
- 📧 ติดต่อ: example@email.com
```

**คนที่ 2** เพิ่มข้อความนี้ (แก้ข้อมูลเป็นของตัวเอง):

```markdown
## สมาชิกคนที่ 2: [ชื่อ-นามสกุล]

- 🎓 รหัสนักศึกษา: 6xxxxxxxx
- 💻 ภาษาโปรแกรมที่ชอบ: JavaScript
- 🎯 งานอดิเรก: ถ่ายรูป, เล่นดนตรี
- 📧 ติดต่อ: example2@email.com
```

บันทึกไฟล์ให้เรียบร้อย

> 🔥 **จุดสำคัญของบทเรียน:** ทั้งสองคนแก้ไฟล์ `README.md` ไฟล์เดียวกัน ตรงตำแหน่งใกล้กัน เมื่อนำมารวมกันในตอนท้าย เราจะได้เรียนรู้วิธีจัดการ **merge conflict** จริง ๆ

---

## ขั้นตอนที่ 6 — บันทึกและส่งงานขึ้น GitHub **[ทั้งคู่]**

ทำตามลำดับ 3 ขั้น: **add → commit → push**

```bash
# 1) ดูว่าไฟล์ไหนถูกแก้ไขบ้าง
git status

# 2) เพิ่มไฟล์เข้าคิวเตรียม commit
git add README.md

# 3) บันทึกการเปลี่ยนแปลงพร้อมข้อความอธิบาย
git commit -m "เพิ่มข้อมูลส่วนตัวของสมาชิก"
```

ส่ง branch ของตัวเองขึ้น GitHub:

**คนที่ 1:**
```bash
git push origin member-a
```

**คนที่ 2:**
```bash
git push origin member-b
```

> ความหมายของแต่ละคำสั่ง:
> - `git add` = หยิบไฟล์ที่แก้ใส่ตะกร้า (staging area)
> - `git commit` = ถ่ายรูปบันทึกสถานะงาน ณ ตอนนั้น พร้อมคำอธิบาย
> - `git push` = อัปโหลด commit ขึ้น GitHub

---

## ขั้นตอนที่ 7 — รวม branch แรกเข้า main **[คนที่ 1]**

ให้ **คนที่ 1** ทำหน้าที่รวมงาน เริ่มจากกลับไป branch `main` แล้วดึงงานล่าสุด

```bash
git checkout main
git pull origin main
```

รวม branch ของคนที่ 1 เข้ามา:

```bash
git merge member-a
```

ครั้งแรกนี้มักจะรวมสำเร็จทันที (ไม่มี conflict) เพราะ `main` ยังไม่มีใครแก้ จากนั้นส่งขึ้น GitHub:

```bash
git push origin main
```

---

## ขั้นตอนที่ 8 — รวม branch ที่สอง และจัดการ Conflict **[คนที่ 1]**

ตอนนี้ลองรวม branch ของคนที่ 2 เข้ามา:

```bash
git merge member-b
```

เนื่องจากทั้งคู่แก้ไฟล์เดียวกัน Git อาจขึ้นข้อความว่า **CONFLICT (content): Merge conflict in README.md**

เปิดไฟล์ `README.md` จะเห็นเครื่องหมายแบบนี้:

```
<<<<<<< HEAD
(เนื้อหาฝั่ง main — ของคนที่ 1)
=======
(เนื้อหาฝั่ง member-b — ของคนที่ 2)
>>>>>>> member-b
```

**วิธีแก้:** ลบเครื่องหมาย `<<<<<<<`, `=======`, `>>>>>>>` ออก แล้วจัดเรียงเนื้อหาให้มีข้อมูลครบทั้งสองคนตามที่ต้องการ เช่น:

```markdown
## สมาชิกคนที่ 1: สมชาย ใจดี
- ...

## สมาชิกคนที่ 2: สมหญิง รักเรียน
- ...
```

บันทึกไฟล์ แล้วยืนยันการแก้ conflict:

```bash
git add README.md
git commit -m "รวมข้อมูลสมาชิกทั้งสองคน และแก้ conflict"
git push origin main
```

🎉 เข้าไปดูที่หน้า GitHub → branch `main` จะมีข้อมูลครบของทั้งสองคนแล้ว!

## สรุปคำสั่งที่ใช้ทั้งหมด

| คำสั่ง | ความหมาย |
|--------|----------|
| `git config --global user.name/email` | ตั้งค่าชื่อและอีเมลผู้ใช้ |
| `git clone <url>` | ดาวน์โหลดสำเนา repo ลงเครื่อง |
| `git status` | ดูสถานะไฟล์ที่เปลี่ยนแปลง |
| `git branch` | ดูรายชื่อ branch ทั้งหมด |
| `git checkout -b <ชื่อ>` | สร้าง branch ใหม่และย้ายเข้าไป |
| `git checkout <ชื่อ>` | สลับไป branch ที่มีอยู่แล้ว |
| `git add <ไฟล์>` | เพิ่มไฟล์เข้าคิวเตรียม commit |
| `git commit -m "ข้อความ"` | บันทึกการเปลี่ยนแปลง |
| `git push origin <branch>` | ส่ง branch ขึ้น GitHub |
| `git pull origin <branch>` | ดึงงานล่าสุดจาก GitHub |
| `git merge <branch>` | รวม branch อื่นเข้า branch ปัจจุบัน |

## แบบฝึกหัดเพิ่มเติม (สำหรับกลุ่มที่ทำเสร็จก่อน)

1. **สลับหน้าที่:** ให้คนที่ 2 ลองสร้าง branch ใหม่ชื่อ `add-photo` แล้วเพิ่มลิงก์รูปโปรไฟล์ของทั้งทีม จากนั้น merge เข้า `main`
2. **ลองใช้ Pull Request:** แทนที่จะ merge ในเครื่อง ให้ลองสร้าง Pull Request บนหน้าเว็บ GitHub แล้วให้เพื่อนกด review และ merge ให้
3. **ดูประวัติ:** ใช้คำสั่ง `git log --oneline --graph --all` เพื่อดูเส้นทางการ merge ของทีมในรูปแบบกราฟ

## ❓ ปัญหาที่พบบ่อย (Troubleshooting)

| อาการ | สาเหตุ / วิธีแก้ |
|-------|----------------|
| `push` ไม่ได้ ขึ้น permission denied | คนที่ 2 ยังไม่ได้กดรับคำเชิญเป็น collaborator |
| เผลอแก้งานบน `main` | ใช้ `git stash` เก็บงานไว้ แล้ว `git checkout -b ชื่อbranch` ค่อย `git stash pop` |
| ลืมว่าตอนนี้อยู่ branch ไหน | พิมพ์ `git branch` ดู branch ที่มี `*` นำหน้า |
| commit ผิดข้อความ | `git commit --amend -m "ข้อความใหม่"` (ก่อน push) |
| อยากยกเลิกการแก้ไฟล์ที่ยังไม่ commit | `git checkout -- <ไฟล์>` |
